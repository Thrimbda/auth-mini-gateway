# Delivery walkthrough: authenticated reverse proxy

> Mode: implementation
> Labels: `high-risk`, `security`
> Review state: verification **PASS**; security review **PASS**
> Compatibility: no schema or gateway cookie-format change

## Reviewer summary

The gateway now has two startup-selected modes:

- **Adapter mode:** leave `UPSTREAM_URL` unset or empty. Existing gateway routes remain compatible and unmatched routes return `404` for nginx `auth_request` deployments.
- **Proxy mode:** set one startup-validated `http`/`https` `UPSTREAM_URL`. Gateway-owned routes remain local; every other safe route is authenticated and streamed to the one fixed upstream.

Both `/auth/check` and proxy fallback use the same session lookup, refresh, policy, identity-safety, and touch decision. Only response mapping differs: `204/401/403/503` for `/auth/check`, and upstream/login `302`/`403`/`503` for proxy fallback.

The handwritten server is replaced by Tokio + Hyper HTTP/1. Proxy bodies flow frame by frame with backpressure. Blocking SQLite, auth-mini, and single-flight operations remain behind bounded 64-active/64-queued admission. A fixed-origin low-level sender pool reuses clean connections without automatically replaying failed requests.

## Trust boundary

- `UPSTREAM_URL` is startup-only, absolute, fixed, and rejects credentials, query, fragment, and unsupported schemes.
- Request Host, headers, cookies, forwarding fields, path, or query cannot replace the configured scheme/authority.
- The six gateway-owned paths are classified before fallback and never reach the app.
- Browser Cookie, Authorization, spoofed `X-Auth-Mini-*`, inbound forwarding, and hop-by-hop fields are stripped.
- External Host is preserved only as app metadata. Forwarded fields are regenerated from local facts.
- Only verified user ID/email are injected. Gateway cookies and auth-mini tokens never reach the app.
- Response gateway-cookie names and malformed `Set-Cookie` fields are dropped; valid repeated application headers remain separate.

## Transport behavior

- Large uploads, chunked responses, SSE, HTTP/1 keep-alive, connection reuse, and backpressure are supported without full body buffering.
- Framing is regenerated by Hyper; trailer frames are intentionally discarded.
- Every request gets one upstream send attempt. A stale pooled connection is not retried.
- WebSocket is the only tunneled upgrade. Both client and upstream handshakes are validated before downstream `101`, then upgraded streams are bridged bidirectionally.
- An upstream early final response cancels remaining upload polling, closes downstream, and prevents reuse of the affected upstream connection.

## Security fixes

| Finding | Resolution |
|---|---|
| Early final upload/reuse ambiguity | Shared upload state cancels later polling, closes downstream, and disables reuse. |
| Non-ASCII identity divergence | Adapter and proxy preserve byte-identical verified header bytes. |
| WebSocket fields nominated by `Connection` | Required request nominations return `400`; invalid upstream `101` nominations return `502` before commitment. |
| HTTP startup required TLS roots | HTTP skips native-root loading; HTTPS remains certificate validated. |

## Deployment and rollback

```text
Proxy:
Browser -> Acorn nginx :443 -> FRP -> gateway 127.0.0.1:7780
        -> OpenCode 127.0.0.1:4096

Adapter rollback:
Browser -> Acorn nginx :443 -> FRP -> node nginx 127.0.0.1:7781
        -> auth_request gateway 127.0.0.1:3000
        -> OpenCode 127.0.0.1:4096 after allow
```

Only one gateway process may use SQLite. Ports `3000` and `4096` must have no public/FRP route. Public TLS remains on Acorn nginx.

Rollback uses maintenance deny: stop proxy `7780`, restart with `UPSTREAM_URL` unset on `3000`, verify retained nginx `7781`, switch FRP from `7780` to `7781`, then remove deny. No data or cookie migration is required.
